## 引言：上下文文件的兴起

随着 AI 编程助手（Coding Agents）的快速普及，一种新的实践正在软件开发社区中流行起来：为代码仓库添加上下文文件，比如 `AGENTS.md` 或 `CLAUDE.md`。这些文件本质上是为 AI 代理准备的"使用说明书"，旨在帮助它们更好地理解项目结构、运行构建命令、遵循代码风格指南等。

各大厂商都在积极推荐这种做法。Anthropic 官方文档建议使用 `CLAUDE.md` 文件来定制化 Claude Code；OpenAI 的 Codex 也支持 `AGENTS.md`；据统计，目前已有超过 60,000 个 GitHub 仓库采用了这种做法。

然而，一个关键问题一直缺乏严谨的研究：**这些上下文文件真的有效吗？**

ETH Zurich 和 LogicStar.ai 的研究团队发表的最新论文《Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?》给出了一个出人意料的答案。

## 研究方法：构建 AGENTBENCH

为了回答这个问题，研究团队需要克服两个主要挑战：

1. **现有基准测试缺乏上下文文件**：SWE-bench 等流行基准中的仓库都没有包含开发者编写的上下文文件
2. **流行仓库不具代表性**：大多数开发者使用的并非是 Django、Flask 这类知名项目

为此，研究团队构建了一个全新的基准测试集 **AGENTBENCH**，包含：

- **138 个独特的任务实例**
- **12 个包含开发者编写上下文文件的仓库**
- 涵盖 bug 修复和功能添加两类任务

这些仓库包括：ragas、smolagents、openai-agents-python、transformers、pr-agent、graphiti、fastmcp、wagtail、opshin、ansible、pdm、tinygrad 等。

## 核心发现：令人意外的结果

### 发现一：LLM 生成的上下文文件降低性能

实验结果令人震惊：**LLM 自动生成的上下文文件在 8 个测试场景中有 5 个降低了任务成功率**。

具体数据：
- SWE-BENCH LITE：成功率平均下降 0.5%
- AGENTBENCH：成功率平均下降 2%
- 推理成本增加：20%-23%

### 发现二：开发者编写的上下文文件仅带来微小提升

即使是开发者精心编写的上下文文件，效果也相当有限：

- 仅在 3/4 的模型上带来性能提升
- 平均提升幅度：约 4%
- 同样会增加推理成本（最高 19%）

### 发现三：上下文文件并非有效的仓库概览

研究团队测量了代理首次接触 PR 修改文件所需的步数，以此评估上下文文件中"仓库概览"部分的效果。结果显示：**无论是 LLM 生成还是开发者编写的上下文文件，都没有显著减少代理发现相关文件所需的时间**。

### 发现四：上下文文件导致更多探索和测试

通过分析代理的工具调用轨迹，研究团队发现上下文文件会导致：

- 运行更多测试（pytest 调用增加）
- 更多的文件搜索和阅读
- 更多使用仓库特定工具（如 uv、repo_tool）
- 需要更多"思考"——推理 token 增加 14%-22%

### 发现五：代理确实会遵循上下文文件中的指令

这不是因为代理"不听话"。实验证明，当上下文文件提到某个工具时：
- `uv` 的使用率从不到 0.01 次提升到 1.6 次
- 仓库特定工具从不到 0.05 次提升到 2.5 次

## 深入分析：为什么上下文文件会降低性能？

### 假设一：冗余信息过多

研究团队做了一个巧妙的实验：在生成上下文文件后，删除仓库中所有的文档文件（.md 文件、示例代码、docs/ 目录）。

在这种"文档荒漠"环境下，LLM 生成的上下文文件反而**持续提升性能 2.7%**，甚至超过了开发者编写的版本。

这解释了为什么一些开发者报告上下文文件有帮助——对于文档匮乏的小众仓库，上下文文件确实有价值。但对于已有完善文档的项目，LLM 生成的上下文文件大多是冗余的。

### 假设二：不必要的指令增加了任务难度

上下文文件通常包含各种"建议"，如"运行测试"、"检查代码风格"等。代理会认真遵循这些指令，但这反而增加了认知负担和执行步骤。

就像给一个有经验的程序员过多不必要的工作指南，反而会干扰他们的正常工作流程。

## 实验细节

研究团队测试了四种主流编程代理：

| 代理 | 底层模型 |
|------|----------|
| Claude Code | Sonnet-4.5 |
| Codex | GPT-5.2 / GPT-5.1 Mini |
| Qwen Code | Qwen3-30B-Coder |

三种测试设置：
- **NONE**：无上下文文件
- **LLM**：使用代理推荐的初始化命令自动生成
- **HUMAN**：使用开发者已编写的上下文文件（仅 AGENTBENCH）

## 实践建议

基于研究发现，论文给出了以下建议：

### 1. 暂时不要使用 LLM 生成的上下文文件

与各厂商的建议相反，LLM 生成的上下文文件目前会降低性能并增加成本。除非你的项目几乎没有文档，否则不建议使用。

### 2. 如果编写上下文文件，保持最小化

开发者编写的上下文文件应该只包含**最基本的必需信息**：
- 项目特定的工具和命令
- 必须遵循的编码规范
- 关键的构建和测试命令

避免添加：
- 冗余的仓库结构描述
- 过于详细的操作指南
- 不必要的最佳实践建议

### 3. 对于文档完善的项目，上下文文件价值有限

如果你的项目已经有良好的 README、CONTRIBUTING 等文档，额外添加上下文文件可能不会带来明显收益。

## 局限性与未来方向

研究团队也指出了本研究的局限性：

1. **仅针对 Python**：对于训练数据中较少出现的小众编程语言，上下文文件可能有更大价值
2. **仅关注任务完成率**：未评估代码安全性、性能等方面的影哏
3. **自动生成方法有待改进**：如何生成更简洁、更相关的上下文文件是一个开放问题

## 总结

这项研究揭示了一个重要的现实：**被广泛推荐的最佳实践不一定真的有效**。

尽管各大 AI 厂商都在大力推广上下文文件，但实证研究表明：
- LLM 自动生成的上下文文件弊大于利
- 开发者手写的上下文文件效果有限
- 上下文文件更像是"冗余文档"而非"有效指南"

这项研究也提醒我们，在 AI 辅助编程这个快速发展的领域，我们需要更多这样严谨的实证研究来验证各种"最佳实践"的真实效果。
