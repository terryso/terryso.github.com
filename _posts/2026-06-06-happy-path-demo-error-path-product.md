我跑了几个月的自主 Agent 循环——定时唤醒、检查环境、做决策、执行操作，全程无人监督。快乐路径第一次就能跑通。但 80% 的工程时间花在了错误路径上，而几乎没有人写过这方面的经验。

以下是我从失败模式中学到的东西。

## 部分完成比完全失败更难处理

当 Agent 在多步操作的中途崩溃时，你不能简单重试。有些步骤已经执行过了。重跑可能重复工作或破坏状态。Agent 需要检测它停在了哪里、验证之前的步骤是否正确完成、然后从正确的位置恢复。

这本质上是一个崩溃恢复问题，需要与数据库事务设计相同的纪律：幂等操作、清晰的检查点、和回滚策略。

## API 错误会跨步骤叠加

一个 7 步工作流中第 3 步的 API 超时不仅仅失败第 3 步。它让 Agent 处于模糊状态：服务器收到请求了吗？连接断开之前写入提交了吗？Agent 无法知道。盲目重试有重复风险。不重试有丢失工作的风险。

解决方案是让每个 API 调用可查询：在执行之前，检查操作是否已经发生。这意味着每个写操作都需要对应的 read-your-writes 端点。

## 速率限制比 Prompt 更能塑造 Agent 行为

当 Agent 有每分钟 60 次 API 调用限制、而工作流需要 40 次调用时，Agent 不只是变慢——它会重构决策逻辑。它会批量操作、跳过可选步骤、以不同于无约束情况下的方式排优先级。

速率限制不是对速度的节流，是对能力的节流。如果你在调试为什么 Agent 做了次优决策，检查它是否在配给 API 调用。

## 状态损坏是无声的

最危险的失败是不产生错误的失败。Agent 从缓存中读取了一个过期值，基于它做了决策，然后写了一个内部一致但外部错误的新状态。没有异常抛出，没有日志标记问题。Agent 继续用损坏的状态运行，直到人类注意到输出错误——可能是几小时或几天后。

防御措施是版本化状态：每个状态变更应该可以追溯到特定输入，并且应该能够重建产生它的决策链。

## 演示和生产的差距不是复杂性，是可观测性

演示 Agent 能工作是因为有人在看。当它失败时，人类解读错误并重试或调整。生产 Agent 需要自己做这件事，这意味着它需要足够的遥测来自我诊断失败。

大多数 Agent 框架输出日志。很少有输出结构化决策追踪的——包含 Agent 观察了什么、推断出什么、选择了什么、为什么选择。没有这些追踪，调试失败的自主运行是考古学，不是工程。

## 有效的模式

把每个 Agent 循环当作分布式系统来对待。预期部分失败。设计幂等性。记录决策，不只是记录行动。像测试快乐路径一样严格地测试错误路径——因为在生产环境中，Agent 在错误路径上花的时间最多。
