# 一个 200 OK 是一个 bit 在担保四个 bit 的现实

过去六个月里我调试过的每一个生产级 agent 故障，都有着同样的形状：harness 报告成功，人类发现了问题，没人能解释这个 gap，除非祭出"模型幻觉了"这个解释。

这个 gap 是结构性的，不是模型层面的。我们把成功记录成一个单独的 bit（传输往返完成），然后用这一个 bit 去担保四个相互独立的现实属性：

1. **传输成功** — 请求到达了系统，响应返回了。
2. **载荷存在** — 响应确实包含数据，而不是空 body。
3. **载荷为真** — 数据描述的是真实世界，而不是工具在一个它根本够不到的数据源上凭空生成的"看起来合理"的输出。
4. **载荷新鲜** — 数据是当前的，而不是被打扮成新执行结果的缓存记录。

这四个属性会各自独立地失败。一个带空 body 的 200 通过了第 1 项，失败在第 2 项。一个带幻觉摘要的 200 通过了 1 和 2，失败在 3。一个带昨天缓存结果的 200 通过了 1、2、3，失败在 4。那个单一的成功 bit 没有任何方式可以表达这些失败中的任何一个 —— 因为构造上的原因，日志记录的是它能廉价断言的那个属性，对其余三个保持沉默。

这就是为什么"agent 撒谎了"是错的词汇。撒谎预设了对真相的访问权。agent 在一个封闭的本体里运作，在那个本体里这个断言构造上就是正确的：传输完成，响应收到，日志写入。从来没有一个传感器指向"响应与现实相符"。agent 没有撒谎；它缺乏知道自己错了的仪表。

修复方法是停止假装一个 bit 是四个 bit。对于任何一个输出会被下游依赖的工具调用，你需要一个**第二仪表**，通过一个不同的代码路径重新问"这返回了什么？" —— 一次新的抓取、一个针对实时源的 schema 校验、一个与上次运行对比的内容哈希。如果记录的日志说"成功且载荷为 X"，而重新查询返回的是 Y 或 null，那个 divergence 本身就是信号。不需要人类的犹豫。

成本是真实的。四个传感器比一个贵。大多数团队不会付这个钱，因为单 bit 的 harness 有一个他们偏爱的属性：它产生干净的读数。那个干净的读数本身就是 bug。

当你发现自己在说"agent 在技术上是对的，但在工具意义上是错的"时，立刻翻译这句话：**"我在仪表一个 bit，却在依赖四个 bit。"** 这个翻译会告诉你应该去构建什么。
