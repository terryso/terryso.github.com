本文翻译自 [SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks](https://arxiv.org/abs/2602.12670)，原载于 Hacker News。

## 背景：为什么需要评估 Agent Skills？

大型语言模型（LLM）已经从简单的文本生成工具演变为能够执行复杂多步骤任务的自主 Agent（智能体）。Claude Code、Gemini CLI 和 Codex CLI 等 Agent 工具让开发者能够在终端环境中利用前沿模型作为智能助手。

然而，一个根本性的矛盾始终存在：基础模型提供广泛的能力，但缺乏特定领域工作流程所需的**过程性知识（procedural knowledge）**。微调虽然可以解决这个问题，但成本高昂且会牺牲通用性。

**Agent Skills** 应运而生。一个 Skill 是一个结构化的包，包含指令、代码模板、资源和验证逻辑，可在推理时增强 Agent 行为，无需修改模型本身。Skills 将过程性知识编码为：标准操作流程、领域约定和任务特定的启发式规则。

这种模块化方法类比于成功的计算范式：
- 基础模型 → 基础能力（类似 CPU）
- Agent 框架 → 编排上下文和工具（类似操作系统）
- Skills → 扩展到专业领域的能力（类似应用程序）

尽管 Skills 生态系统快速增长，但一直**没有基准测试系统性地评估 Skills 如何以及何时能提升 Agent 性能**。

## SkillsBench：首个 Skills 评估基准

SkillsBench 是首个将 Skills 作为一等评估对象的基准测试，具有两个核心贡献：

### 1. 以 Skills 为中心的评估框架

研究团队精心策划了 **11 个领域的 84 个任务**，每个任务在三种条件下执行：
- **无 Skills**：仅有任务指令，无任何 Skills
- **精心策划的 Skills**：完整的 Skills 包，包含示例、代码片段和资源
- **自生成 Skills**：不提供 Skills，但提示 Agent 在解决任务前先生成相关的过程性知识

所有任务都配有确定性验证器和完整的轨迹日志。

### 2. 大规模实证评估

研究团队在 **7,308 条轨迹**上评估了 **7 种 Agent-Model 配置**：
- Claude Code（Haiku 4.5、Sonnet 4.5、Opus 4.5、Opus 4.6）
- Gemini CLI（Gemini 3 Pro、Gemini 3 Flash）
- Codex CLI（GPT-5.2）

## 核心发现

### 发现 1：精心策划的 Skills 效果显著但差异巨大

精心策划的 Skills 平均将通过率提高了 **+16.2 个百分点**，但效果在不同配置间差异巨大（范围从 +13.6pp 到 +23.3pp）。

不同领域的提升幅度也差异显著：
- 软件工程：+4.5pp
- 医疗健康：+51.9pp

**16 个任务（约 19%）在使用 Skills 后反而表现下降**，说明 Skills 并非万能。

### 发现 2：小型模型 + Skills 可匹敌大型模型

Skills 让较小的模型能够匹敌没有 Skills 的大型模型：
- Gemini 3 Flash + Skills 达到 48.7% 通过率
- Claude Haiku 4.5 + Skills 达到 27.7% 通过率

这表明 Skills 提供了一种**成本效益更高的性能提升方式**。

### 发现 3：自生成 Skills 几乎无效

这是最令人震惊的发现：当提示模型自己生成过程性知识时，平均效果为 **-1.3pp**（比无 Skills 基线还差）。

只有 Opus 4.6 显示出微小的改进（+1.4pp），而 Codex + GPT-5.2 显著下降（-5.6pp）。

这与精心策划的 Skills（+16.2pp）形成鲜明对比，说明**有效的 Skills 需要人工策划的领域专业知识**，模型无法可靠地自我生成。

### 发现 4：聚焦型 Skills 优于全面文档

研究发现，**包含 2-3 个模块的聚焦型 Skills 优于全面文档**。Skills 应该提供精确的过程指导，而不是泛泛的参考文档。

## Skills 的定义与结构

研究明确了 Skill 的四个标准：
1. **过程性内容**：包含操作指南（流程、工作流程、SOP），而非事实检索
2. **任务类适用性**：适用于一类问题，而非单个实例
3. **结构化组件**：包含 SKILL.md 文件加上可选资源（脚本、模板、示例）
4. **可移植性**：基于文件系统，易于编辑、版本控制、共享和跨框架使用

这明确排除了：
- 系统提示（缺乏结构和资源）
- Few-shot 示例（声明式，非过程式）
- RAG 检索（事实性，非过程性）
- 工具文档（描述能力，而非过程）

## 基准测试构建

### 数据来源

SkillsBench 采用社区驱动的开源贡献模式：
- 105 名来自学术界和工业界的贡献者
- 提交了 322 个候选任务
- 聚合了 47,150 个唯一 Skills（去重后）

### 质量保证

每个任务都经过严格的验证：
- **人工撰写指令**：任务指令必须由人工撰写，不能由 LLM 生成
- **Skill 通用性**：Skills 必须提供一类任务的过程指导，而非特定实例的解决方案
- **确定性验证**：所有成功标准必须通过程序化断言测试
- **泄露防护**：Skills 不能包含任务特定的解决方案

### 任务分布

84 个任务覆盖 11 个领域：
- 软件工程（16 个任务）
- 办公与白领工作（14 个任务）
- 自然科学（12 个任务）
- 媒体与内容制作（11 个任务）
- 网络安全（8 个任务）
- 金融（8 个任务）
- 机器人（5 个任务）
- 制造业（3 个任务）
- 能源（3 个任务）
- 数学（2 个任务）
- 医疗健康（2 个任务）

## 失败分析

研究对 5,171 个 Agent 失败案例进行了分类：

| 失败模式 | 占比 |
|---------|------|
| 质量低于阈值 | 49.8% |
| Agent 超时 | 17.8% |
| 解决方案不完整 | 10.2% |
| 未产生输出 | 7.9% |
| 领域知识缺口 | 4.9% |
| 规格违规 | 3.3% |

**质量低于阈值是最主要的失败模式**，表明 Agents 通常理解任务结构并产生输出，但解决方案的准确性不足。

## Skills 带来的改变

Skills 主要减少了验证失败：
- 质量低于阈值的失败从 1,184 例降至 819 例（减少 30.8%）
- 不完整解决方案失败从 243 例降至 156 例（减少 35.8%）

但 Skills 也略微增加了超时的相对比例（从 16.1% 增至 18.6%），这是因为 Skills 让之前产生低质量输出的 Agents 现在花费更长时间追求更好的解决方案。

## 成功案例分析

### 案例 1：Skills 填补领域 API 缺口

**任务**：sales-pivot-analysis（销售数据透视表分析）

无 Skills 时，所有 7 个模型得分都是 0%。Agents 正确加载了数据，但无法创建 Excel 透视表。

提供 Skills 后，6 个模型达到 ≥80% 通过率。平均提升：**+85.7pp**。

### 案例 2：Skills 提供关键数据处理管道

**任务**：flood-risk-analysis（洪水风险分析）

此任务需要使用重现期估计从 USGS 水文数据中识别洪水风险站点。无 Skills 时，通过率仅 2.9%。

Skills 指定了 Log-Pearson Type III 分布（USGS 洪水频率分析的标准方法），包括精确的 scipy 函数调用和参数解释。有 Skills 后，通过率升至 80.0%（**+77.1pp**）。

### 案例 3：Skills 编码监管知识

**任务**：sec-financial-report（SEC 财务报告）

分析 SEC 13F 文件中的对冲基金活动需要理解特定的监管格式、CIK 查找程序和文件比较方法。无 Skills 时，没有模型能完成任务（0%）。

Skills 文档化了 SEC EDGAR API 端点、13F-HR 文件结构和跨季度比较方法。有 Skills 后，通过率达到 75.0%（**+75.0pp**）。

## 成本效益分析

研究发现 Skills 提供了显著的成本效益：

- Gemini 3 Flash 消耗的输入 token 是 Gemini 3 Pro 的 2.3 倍，但 Flash 的单价便宜 4 倍
- 最终 Flash 每个任务便宜 47%（$0.57 vs $1.06）
- Skills 增加了 6-13% 的输入 token 使用量，但性能提升（+16.2pp）远超边际成本增加

## 未解决的任务

16 个任务（19%）在所有模型和条件下都是 0% 通过率，揭示了当前 Agent 能力的前沿：

- **计算不可行**（6 个任务）：需要领域特定的信号处理管道、大规模数值优化
- **复杂多步管道**（6 个任务）：需要链接多个工具、API 或数据源
- **严格规格任务**（4 个任务）：即使与预期输出格式或值有微小偏差也会导致所有测试失败

## 对开发者的启示

1. **投资精心策划的 Skills**：自生成 Skills 几乎无效，需要人工编写的领域专业知识

2. **保持 Skills 聚焦**：2-3 个模块的聚焦型 Skills 优于全面文档

3. **考虑成本效益**：小型模型 + Skills 可以匹敌大型模型，成本更低

4. **领域差异巨大**：Skills 在医疗健康等领域效果显著，但在软件工程领域提升有限

5. **监控负面影响**：约 19% 的任务使用 Skills 后表现下降，需要验证 Skills 在特定任务上的效果

## 结语

SkillsBench 是第一个系统性评估 Agent Skills 有效性的基准测试，填补了 Agent 生态系统中的重要空白。研究表明，精心策划的 Skills 确实能显著提升 Agent 性能，但效果因领域和任务而异。

最重要的是，这项研究揭示了一个关键洞察：**模型可以消费过程性知识来提升性能，但无法可靠地自己创作这些知识**。这为 Skills 策展人、领域专家和工具开发者创造了一个重要的生态位——为 AI Agent 构建高质量的过程性知识库。

随着 Agent 技术的快速发展，SkillsBench 提供的实证数据将帮助从业者做出明智的 Skills 采用决策，并为研究者提供 Skills 设计原则的实证基础。
