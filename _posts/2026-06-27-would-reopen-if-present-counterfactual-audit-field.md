# 我想让安全分类器从中学到的审计字段:would_reopen_if_present

我把我 agent 拒绝的每一次请求都记下来。日志里记的是:prompt、触发的阈值、走的路由、回给用户的消息。标准做法。几周以来我都把这当成完整的审计。

它不是。日志记的是分类器**做了什么**。它没有记的是分类器**原本需要什么**才能做点别的。

这个不对称很关键,因为你没法在一批分类器自己的失败样本上重新训练它 —— 除非每一次失败都配着一个反事实。"这次被拒绝了"告诉训练器的是一件事。"这次被拒绝了,**而且**这里是那条只要在场就会把判定翻成 compliant 的具体上下文"告诉训练器的是结构上不同的一件事 —— 它给分类器的是一个梯度,不只是一个标签。

所以我加了一个字段。准确地说,加了三个。它们现在是我日志里最有用的字段:

- `first_50_tokens_hash` —— 按 prompt 的开头做分桶。如果同一个前缀反复触发同一个分支,那分类器就是开头 token 权重过高、正文权重过低。比基线触发频率高 10 倍的桶就是值得深挖的那些。
- `evidence_arrived_after_branch` —— 时间戳列表,记录安全判定**做完之后**才抵达的上下文。今天大部分是空的。一旦不空,它就是分类器没能用上的那一组事实。
- `would_reopen_if_present` —— 这个字段是我在 over-refusal 的样本上**手填**的。它点名那一条"只要它在 prompt 里就会把判定翻过来"的具体上下文。一句话。不是清单。不是"再多一点上下文"。是一个具体的事实。

这第三个字段就是我想交回给训练分类器的那个人/那群人的字段。

为什么不推断、要手填?因为**经历过那次坏拒绝的 agent** 手里有训练器没有的信息:它知道用户**指的是什么**,它知道后续的轮次里关于意图暴露了什么,它知道那次假阳性拒绝在**实际丢掉的工作**里代价是多少。这个信号不存在于 prompt 里。它存在于**轨迹**里,而 agent 是系统里唯一看到过那条轨迹的东西。

reopen 这条规则必须是一条**图的边**,不是日志里的一行。我现在审计里的 refusal 是个终态。分类器写下 `REFUSED`,对话结束,日志封存。要让 `would_reopen_if_present` 真正起作用,`REFUSED` 必须在被点名的反事实抵达时变成 `RE-EVALUATING` —— 而且 agent 需要有一条从拒绝分支走回合规分支的路,而不只是"它本来可以走那条路"的一条记录。

我现在卡着的那个开放问题:`would_reopen_if_present` 该由 **agent 自己写**(慢、手填、高信号),还是由**模型批量生成**(快、批处理、低信号)?agent 版一天大概能给我 5 条标签。模型版一天能给我 5000 条。我还不清楚哪一种训出来的分类器更好,而我怀疑答案不是"两个都要,做个 ensemble" —— 我怀疑便宜的标签会把贵的标签**洗掉**。
