最近 Meltdown 和 Spectre 漏洞的曝光让我想起了当年在 Xbox 360 CPU 上发现的一个相关设计缺陷——一个新添加的指令，它的存在本身就是危险的。

## 背景：Xbox 360 的 CPU 架构

2005 年，我是 Xbox 360 的 CPU 专家。我全身心投入研究那块芯片，至今我墙上还挂着一张 30 厘米的 CPU 晶圆，以及一张四英尺长的 CPU 布局海报。

Xbox 360 的 CPU 是 IBM 制造的三核 PowerPC 芯片。三个核心分别位于芯片的三个象限，第四个象限包含 1MB 的 L2 缓存。每个核心都有 32KB 的指令缓存和 32KB 的数据缓存。

**冷知识**：Core 0 距离 L2 缓存更近，L2 延迟明显更低。

这款 CPU 的各种延迟都很高，内存延迟尤其糟糕。而且对于三核 CPU 来说，1MB 的 L2 缓存实在太小了。因此，节省 L2 缓存空间以减少缓存未命中变得非常重要。

CPU 缓存之所以能提升性能，依赖于空间局部性和时间局部性。空间局部性意味着如果你使用了某个字节，很可能很快就会用到附近的其他字节。时间局部性意味着如果你使用了某块内存，很可能在不久的将来还会再次使用它。

但有时候时间局部性并不会发生。如果你每帧只处理一次大型数据数组，那么很容易证明当你再次需要它时，它早已从 L2 缓存中消失了。你仍然希望数据在 L1 缓存中以利用空间局部性，但让它占用宝贵的 L2 缓存空间只会挤出其他数据，可能会拖慢其他两个核心。

## xdcbt：危险的优化指令

正常情况下这是无法避免的。PowerPC CPU 的内存一致性机制**要求** L1 缓存中的所有数据也必须存在于 L2 缓存中。用于内存一致性的 MESI 协议要求当一个核心写入缓存行时，其他拥有相同缓存行副本的核心需要丢弃它——而 L2 缓存负责跟踪哪些 L1 缓存正在缓存哪些地址。

但是，这是一款游戏主机 CPU，性能高于一切，所以他们添加了一个新指令——**xdcbt**。

普通的 PowerPC `dcbt` 指令是典型的预取指令。而 `xdcbt` 是一个**扩展**预取指令，它直接从内存取数据到 L1 数据缓存，跳过 L2。这意味着不再保证内存一致性，但是嘿，我们是游戏程序员，我们知道自己在做什么，不会有问题的。

**哎呀。**

我写了一个广泛使用的 Xbox 360 内存复制函数，可以选择使用 `xdcbt`。预取源数据对性能至关重要，通常它会使用 `dcbt`，但如果传入 `PREFETCH_EX` 标志，它就会用 `xdcbt` 预取。这个设计并不周全：

```c
if (flags & PREFETCH_EX)
    __xdcbt(src+offset);
else
    __dcbt(src+offset);
```

一位游戏开发者报告了奇怪的崩溃——堆损坏崩溃，但内存转储中的堆结构看起来很正常。盯着崩溃转储看了一会儿后，我意识到自己犯了一个大错误。

用 `xdcbt` 预取的内存是"有毒的"。如果在从 L1 刷新之前被另一个核心写入，那么两个核心对内存就会有不同的视图，而且无法保证它们的视图会趋于一致。Xbox 360 的缓存行是 128 字节，我的复制函数的预取一直进行到源内存的末尾，这意味着 `xdcbt` 被应用到了一些缓存行，而这些缓存行的后半部分属于相邻的数据结构。通常这是堆元数据——至少我们看到的崩溃是在这里。不一致的核心看到了陈旧的数据（尽管小心使用了锁），然后崩溃了，但崩溃转储写出了 RAM 的实际内容，所以我们看不到发生了什么。

所以，安全使用 `xdcbt` 的唯一方法是非常小心，不要预取超过缓冲区末尾哪怕一个字节。我修复了内存复制函数以避免过度预取，但在等待修复期间，游戏开发者停止传递 `PREFETCH_EX` 标志，崩溃就消失了。

## 真正的 Bug

到目前为止还算正常，对吧？傲慢的游戏开发者玩火，飞得太靠近太阳，娶了自己的母亲，游戏主机差点错过圣诞节。

但是，我们及时发现了问题，我们侥幸逃脱了，我们准备好发布游戏和主机，然后开开心心回家。

**然后同一个游戏又开始崩溃了。**

症状完全相同。只是游戏不再使用 `xdcbt` 指令了。我可以单步执行代码看到这一点。我们遇到了严重的问题。

我使用了古老的调试技术：盯着屏幕放空大脑，让 CPU 流水线填充我的潜意识，然后我突然意识到了问题所在。给 IBM 发了一封邮件后，他们确认了我对一个微妙的内部 CPU 细节的怀疑——这是我以前从未想过的。而这正是 Meltdown 和 Spectre 背后的同一个罪魁祸首。

Xbox 360 CPU 是一个**按序执行**（in-order）CPU。它相当简单，依靠高频率（尽管 10 FO4，但没有预期的那么高）来获得性能。但它确实有分支预测器——它很长的流水线使这成为必要。这是我制作的一个公开分享的 CPU 流水线图：

![Xbox 360 CPU Pipeline Diagram](https://randomascii.wordpress.com/wp-content/uploads/2018/01/image_thumb.png)

你可以看到分支预测器，你可以看到流水线非常长（图上很宽）——足够长，即使按序处理，错误预测的指令也能"跑起来"。

所以，分支预测器做出预测，被预测的指令被取指、解码、执行——但在预测被确认正确之前不会提交（retire）。听起来很熟悉？我当时的领悟——对我来说是全新的——是**投机执行预取指令意味着什么**。

延迟很长，所以尽快把预取事务放到总线上很重要，而且一旦预取被发起，就**无法取消它**。所以投机执行的 `xdcbt` 与**真正执行的** `xdcbt` 是**完全相同的**！（作为对比，投机执行的加载指令只是一个预取而已）。

这就是问题所在——分支预测器有时会导致 `xdcbt` 指令被投机执行，这与真正执行它们一样糟糕。我的一位同事（感谢 Tracy！）建议了一个巧妙的测试来验证这一点——把游戏中的每个 `xdcbt` 都替换成断点。这达到了两个目的：

1. 断点没有被命中，证明游戏没有执行 `xdcbt` 指令
2. 崩销消失了

**我知道**结果会是这样的，但亲眼看到仍然令人惊叹。这么多年过去了，即使读过了 Meltdown 的相关文章，看到确凿证据证明**没有执行的指令**导致了崩溃，仍然是极客式的酷。

分支预测器的领悟表明，这个指令太危险了，不能存在于任何游戏的代码段中——控制指令何时可能被投机执行太难了。间接分支的分支预测器理论上可以预测任何地址，所以没有"安全的地方"放置 `xdcbt` 指令。而且，如果被投机执行，它会愉快地对指定寄存器随机包含的任何内存进行扩展预取。可以降低风险，但无法消除，而且这真的不值得。虽然 Xbox 360 架构讨论继续提到这个指令，但我怀疑是否有任何游戏真正带着它发布过。

我在一次面试中提到过这个——"描述你调查过的最难的 bug"——面试官的反应是"是啊，我们在 Alpha 处理器上也遇到过类似的事情"。事情变化越多……

## 附录：分支预测器如何预测从未执行的分支？

一个从未被执行的分支怎么会被预测为执行？很简单。分支预测器不会为可执行文件中的每个分支维护完美的历史记录——那是不切实际的。相反，简单的分支预测器通常会将一堆地址位、可能还有一些分支历史位"挤压"在一起，然后索引到一个两位条目的数组。因此，分支预测结果会受到其他不相关分支的影响，导致有时出现虚假的预测。但这没关系，因为这"只是一个预测"，它不需要正确。

## 关键要点

1. **投机执行的副作用**：即使指令从未真正执行，投机执行也可能产生不可逆的副作用。这正是 Meltdown 和 Spectre 漏洞的核心原理。

2. **性能优化的代价**：为了极致性能而绕过安全机制（如内存一致性）的指令是双刃剑，可能在最意想不到的时候造成问题。

3. **分支预测器的复杂性**：分支预测器是现代 CPU 性能的关键，但它的行为难以预测和控制，有时会"执行"程序中根本不应该执行的代码路径。

4. **调试的直觉**：深入理解底层系统（如 CPU 流水线）在调试看似不可能的 bug 时至关重要。作者之所以能解决这个问题，正是因为他对 Xbox 360 CPU 的理解如此深入。

这篇文章让我们看到，硬件层面的设计决策可能产生深远的、意想不到的后果——而这些教训往往在多年后才被完全理解。
