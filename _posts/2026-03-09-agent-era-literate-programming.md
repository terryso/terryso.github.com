## 什么是文学化编程？

**文学化编程（Literate Programming）** 是一个核心理念：代码应该与散文式的解释交织在一起，使得一个不了解背景的读者也能像阅读故事一样阅读代码库，并理解它的工作原理和功能。

这个概念并不新鲜，早在 1980 年代 Donald Knuth 就提出了这一思想。但在实践中，文学化编程一直面临着巨大的障碍。

## 历史困境：双重叙事的维护负担

尽管我一直对这个理念很着迷，也在几个不同的场景下使用过，但我发现文学化编程在实际操作中会变成一个繁重的任务：**你需要维护两套平行的叙事体系**——代码本身和解释性文字。这显然限制了它的广泛采用。

目前，文学化编程最常见的应用场景是数据科学社区的 **Jupyter Notebooks**，在那里，解释与计算过程及其结果一起呈现在浏览器中。

熟悉 Emacs 的读者可能知道，**Org Mode** 通过 `org-babel` 包支持多语言的文学化编程，允许在文档中执行任意语言并将结果捕获回文档中。但这始终是一个小众模式，主要吸引像我这样的"技术宅"。

## Org Mode 的痛点

即使对于像我这样热衷于这种模式的人来说，将 Org Mode 作为大型软件项目的真实来源（source of truth）也变得笨重：

- 源代码本质上变成了编译输出
- 每次 Org 文件编辑后，代码必须重新提取（在 Org Mode 术语中称为"tangle"）并放置到目标位置
- 虽然这可以自动化，但很容易陷入麻烦的情况：你或你的 agent 编辑了实际源代码，然后在下一次 tangle 时被覆盖

## Agent 时代的转机

但是，**有了编程 Agent 之后，一切都不同了**。

Claude、Kimi 等 LLM 都非常擅长理解 Org Mode 语法。这是一个宽容的标记语言，而 LLM 在这方面表现出色。所有文档都在线可用，很可能已经在训练数据中。虽然 Org Mode 的一个缺点是语法太多，但对于语言模型来说，这完全不是问题。

### 实际应用场景

现在，当我想要测试一个功能时，我会让 Agent 用 Org 格式写一个 **runbook（操作手册）**：

1. **审阅阶段**：散文部分解释了模型对每一步意图的思考，代码块可以在我审阅完成后交互式执行
2. **执行阶段**：可以逐个执行或像脚本一样执行整个文件
3. **结果存储**：执行结果会存储在文档中代码块的下方，就像 Jupyter notebook 一样

### 维护负担的消失

最令人兴奋的是：

- 我可以编辑散文，让模型更新代码
- 或者编辑代码，让模型将含义反映到文本上
- 或者让 Agent 同时修改两者

**维护两个平行系统的问题就这样消失了。**

## 技术实现

Agent 可以被指示处理 tangle 过程，提取问题也随之解决。你可以通过 `AGENTS.md` 文件指示 Agent：

- 将 Org Mode 文件作为真实来源
- 始终用散文解释发生了什么
- 执行前先进行 tangle

Agent 非常擅长所有这些事情，而且它永远不会厌倦在代码微调后重新用散文解释。

## 为什么这很重要？

文学化编程的根本额外劳动——我相信这也是它没有被广泛实践的原因——**被 Agent 消除了**，而且这正好利用了 LLM 最擅长的能力：翻译和总结。

### 额外的好处

1. **多格式导出**：代码库现在可以导出为多种格式，便于舒适地阅读。如果工程师的主要角色正在从编写转向阅读，这一点尤为重要。

2. **代码质量提升**：虽然没有数据支持，但我怀疑文学化编程会提高生成代码的质量，因为解释每个代码块意图的散文会与代码本身一起出现在上下文中。

## 我的实践

我个人还没有机会在更大、更严肃的代码库上尝试这种模式。到目前为止，我只在工作流中使用这种方式进行测试和记录手动流程，但我对其在那里应用感到非常兴奋。

## 格式的限制

我也认识到 Org 格式是一个限制因素，因为它与 Emacs 的紧密集成。然而，我一直认为 Org 应该脱离 Emacs。

我更愿意推广像 **Markdown** 这样的格式，但 Markdown 缺乏包含元数据的能力。相比之下，Org Mode 有属性（properties）的概念，允许从 Emacs Lisp 对文档进行程序化操作。

有趣的是，过去这意味着我经常被诱惑去折腾一会儿 Lisp 来获得想象中的交互功能，但我实际上从未有时间。但现在，LLM 会很乐意把一些 Emacs Lisp 塞进文件的变量部分，为那个特定的交互文档提供定制功能。

## 结语

这不是 Emacs 的具体实现让我兴奋，而是 **Org 对文学化编程的实现所体现的理念本身**。

**有了 Agent，拥有可以像叙事一样阅读的大型代码库——其散文由不知疲倦的机器与代码变化保持同步——是否变得实用？**

我认为这是一个令人深思的问题。

---

## 关键要点

1. **文学化编程的核心价值**：让代码像故事一样可读，降低理解门槛
2. **历史障碍**：维护代码与文档的双重负担太大
3. **Agent 的革命性影响**：AI 可以自动同步代码和文档，消除维护负担
4. **实际应用**：适合测试文档、操作手册等场景
5. **格式选择**：Org Mode 功能强大但与 Emacs 绑定，Markdown 更通用但缺乏元数据支持
6. **未来展望**：随着 Agent 能力提升，文学化编程可能迎来复兴

对于我们中国开发者来说，这个思路特别值得思考——在 AI 辅助编程时代，我们是否应该重新审视那些曾经因为"太麻烦"而被放弃的优秀理念？
