## 好消息来了

CPython JIT 项目提前达成了我们的性能目标！在 macOS AArch64 上提前了一年多，在 x86_64 Linux 上也提前了几个月。

具体数据如下：
- **macOS AArch64**：比尾调用解释器（tail calling interpreter）快 **11-12%**
- **x86_64 Linux**：比标准解释器快 **5-6%**

需要注意的是，这些数字是几何平均数，目前还是初步结果。实际性能范围从 **20% 的减速到超过 100% 的加速** 都有可能（排除 `unpack_sequence` 微基准测试）。另外，我们还没有完全支持 free-threading（自由线程），但目标是在 3.15/3.16 版本中实现。

**JIT 现在重回正轨了。**

## 这一路走来有多难

**我真的无法形容这有多艰难。** 有一段时间，我甚至认真怀疑过 JIT 项目是否真的能带来有意义的性能提升。

回顾一下：最初的 CPython JIT 几乎没有任何性能提升。8 个月前，我写了一篇关于 JIT 反思的文章，讲述了 Python 3.13 和 3.14 中的 JIT 经常比解释器还慢。那时也正是 Faster CPython 团队失去主要赞助商资金的时候。作为一名志愿者，这对我影响不大，但更重要的是，这确实影响了我在那里的朋友们。一度，JIT 的未来看起来很不确定。

## 从 3.13 和 3.14 到现在，发生了什么变化？

我不想把这事讲成什么英雄救美的故事，好像我们凭智慧把 JIT 从失败的边缘拯救回来一样。老实说，我把我们目前的成功很大程度上归功于 **运气**——对的时间、对的地点、对的人、对的赌注。

我真心认为，如果缺少了任何一位核心 JIT 贡献者，这一切都不可能实现：Savannah Ostrowski、Mark Shannon、Diego Russo、Brandt Bucher，还有我。同样感谢其他活跃的 JIT 贡献者：Hai Zhu、Zheaoli、Tomas Roun、Reiden Ong、Donghee Na（可能还有我遗漏的人）。

在这篇文章里，我想聊聊 JIT 项目中较少被提及的部分：**人和运气**。如果你想了解我们是如何做到的技术细节，可以看[这里](https://fidget-spinner.github.io/posts/jit-reflections.html)。

## 第一部分：社区管理

Faster CPython 团队在 2025 年失去了主要赞助商。我立即提出了社区托管的想法。当时，我并不确定这能否成功。JIT 项目向来不以适合新贡献者著称，它历史上需要大量的专业知识。

在剑桥举行的 CPython 核心冲刺（Core Sprint）期间，JIT 核心团队会面并制定了计划：3.15 版本实现 5% 的性能提升，3.16 版本实现 10% 的提升，同时支持 free-threading。

一个不那么引人注目但对项目健康至关重要的目标是：**降低「公交车系数」（bus factor）**。我们希望在 JIT 的所有 3 个阶段都有至少 2 名活跃维护者：
- 前端（region selector）
- 中端（optimizer）
- 后端（code generator）

此前，JIT 的中端只有 2 名活跃的持续贡献者。如今，JIT 的中端有 4 名活跃的持续贡献者，我认为其中 2 位非核心开发者（Hai Zhu 和 Reiden）是有能力且受重视的成员。

### 什么方法有效？

**把复杂问题分解成可管理的小任务**。Brandt 在 3.14 版本期间就开始这样做了，他创建了多个「超级 issue」，把 JIT 优化拆分成简单任务。比如我们会说「尝试优化 JIT 中的单个指令」。

我在 3.15 版本中延续了 Brandt 的想法。幸运的是，我的任务更容易一些，因为我的 issue 涉及将解释器指令转换为易于优化的形式。为了鼓励新贡献者，我还提供了非常详细的、可立即执行的说明，并清晰地划分了工作单元。

我相信这确实有帮助——我们有 11 位贡献者（包括我）在那个 issue 上工作，几乎将整个解释器转换为更友好的 JIT 优化形式。

**核心在于**：JIT 可以从一个不透明的庞然大物，变成一个没有 JIT 经验的 C 程序员也能贡献代码的项目。

其他有效的方法包括：鼓励人们、庆祝大大小小的成就。每个 JIT PR 都有明确的结果，我相信这给了人们方向感。

## 第二部分：幸运的赌注

### 追踪记录（Trace Recording）

再次强调，我把这很大程度上归功于运气。在剑桥的 CPython 核心冲刺期间，Brandt 用一个有趣的技术挑战「勾引」我把 JIT 前端重写为追踪式的。

起初我不喜欢这个想法，但出于一种友好的「较劲」心态，我想先做出来，证明给他看这行不通。

初始原型在 3 天内完成了，但花了一个月才让它正确地运行 JIT 而不失败测试套件。最初的结果很惨淡——在 x86_64 Linux 上慢了约 6%。我正准备放弃这个想法，直到一个幸运的意外发生了：**我误解了 Mark 的建议**。

Mark 之前建议将调度表（dispatch table）通过解释器传递，从而在解释器中有两个调度表（一个是普通解释器，一个是用于追踪的）。Mark 建议我们应该让追踪表中的指令是普通指令的追踪版本。

然而，我理解错了，想出了一个更极端的版本：不是普通指令的追踪版本，而是只有一个指令负责追踪，第二个表中的所有指令都指向它。

**这竟然是一个非常好的选择。**

我发现最初的双表方法之所以慢得多，是因为解释器的体积翻倍了，导致编译后的代码膨胀，自然就慢了。通过只使用单个指令和两个表，我们只增加了解释器 1 条指令的大小，同时保持了基础解释器超快的速度。我亲切地称这个机制为 **dual dispatch（双调度）**。

追踪记录解释器的设计还有很多内容。说句自夸的话，我真的认为它是微型艺术品。我花了一周时间迭代，直到它整体变快。从慢 6% 到使用双调度后几乎没有性能变化。之后，我又解决了追踪解释器中一堆慢的边缘情况，最终让它快了 1.x%。

追踪解释器本身只比专门化解释器慢 3-5 倍（根据我的估计）。关键在于它尊重专门化解释器的所有正常行为，并且基本不干扰它。

给你一个概念：**追踪记录让 JIT 代码覆盖率提高了 50%**。这意味着所有未来的优化可能会比原来有效约 50%（假设所有代码执行相同，当然这不是真的，请多多包涵 :)。

所以我必须感谢 Brandt 和 Mark，让我偶然发现了这么好的解决方案。

### 引用计数消除（Reference Count Elimination）

我们早期做的另一个幸运赌注是尝试引用计数消除。这是 Matt Page 最初在 CPython 字节码优化器中做的工作（更多细节在之前的[优化博客文章](https://fidget-spinner.github.io/posts/jit-reflections.html)中）。

我注意到，即使有了字节码优化器的工作，JIT 代码中每个引用计数减少操作仍然有一个分支。我想：「为什么不尝试消除这个分支呢？」，完全不知道这会有多大帮助。

结果发现，**单个分支实际上相当昂贵**，而且这些会随着时间累积。特别是如果每个 Python 指令都有 >=1 个分支！

另一个幸运的部分是这非常容易并行化，而且它是教人们了解解释器和 JIT 的绝佳工具。这是我们在 Python 3.15 JIT 中引导人们工作的主要优化。虽然这主要是一个手动重构过程，但它教给了人们了解 JIT 所需的关键部分，而不会让他们感到不知所措。

## 第三部分：优秀的团队

我们有一个优秀的基础设施团队。我半开玩笑地说这个，因为它是 **一个人**。实际上，我们的「团队」目前是 Savannah 衣柜里运行的 4 台机器。

尽管如此，Savannah 为 JIT 做了相当于整个基础设施团队的工作。如果我们没有地方报告性能数据，JIT 不可能进展得这么快。**每日 JIT 运行在反馈循环中改变了游戏规则**。它帮助我们捕捉 JIT 性能的回归，并让我们知道我们的优化确实有效。

Mark 技术上非常出色，我觉得互联网给他的赞扬已经够多了，所以我就不多说了 :).

Diego 也很棒。他负责 ARM 硬件上的 JIT，最近还开始让 JIT 对分析器（profiler）更友好。我无法形容这个问题有多难。

Brandt 为我们的机器码后端奠定了基础，没有他，新贡献者就要写汇编代码了，这可能会吓跑更多人。

## 总结

Python 3.15 的 JIT 能够重回正轨，关键在于：

1. **社区协作**：将复杂的 JIT 优化工作分解成小任务，让没有 JIT 经验的开发者也能参与贡献
2. **技术突破**：
   - 追踪记录（Trace Recording）让 JIT 代码覆盖率提高了 50%
   - 引用计数消除（Reference Count Elimination）大幅减少了分支开销
3. **基础设施**：每日性能测试帮助快速发现回归和验证优化效果
4. **人际网络**：与编译器社区保持交流，学习其他项目（如 PyPy）的经验

**人是重要的，再加上一些运气，JIT 就能飞速运转。**
