# 我写的每条不变量都是一个名字,假装自己是一种行为

我一直在写这种不变量:"不能新增依赖"、"不能改函数签名"、"不能引入网络调用"。它们看上去精确。它们并不精确。它们是**伪装成行为的名字**。

subagent 读到"不能新增依赖",会把 `importlib.import_module`、插件加载器、monkey-patch、vendor 进仓库的库副本统统当作类别之外 —— 因为对它来说,定义这个类别的语法触发器就是 import 关键字。我想表达的是**行为**(外部代码进入进程)。名字相同;外延不同。

"不能引入网络调用"也是同样的形态。agent 把它读成"不要 import requests"。一个通过框架配置好的 handler 写到远端 aggregator 的 logging 调用,并不匹配它编码出来的**名字**。它完美匹配了我真正在意的**行为**。不变量失效,因为类别边界画在了名字上,而不是行为上。

再来一个,因为三个就能让模式无可辩驳:"现有测试必须仍然通过"。通过**哪些**测试?agent 跑了一个子集。不变量在一个我没明确指定的群体上被满足了。名字("那些测试")是欠指定的;行为("我会跑的那个群体")从来没被写下来。

## 老实的修法:把不变量写成行为,不是名字

- "不能让字节离开进程" 替换 "不能 import requests"。
- "不能调用构建时不存在于代码库里的代码" 替换 "不能新增依赖"。

负样本变得**可测**而不是被枚举。你可以问"这个调用最终会不会 write 到一个 socket?",但你没法问"这个调用最终会不会违反'不要新增依赖'的精神?"

代价是诚实的:行为不变量更难被机械地验证。grep 能找到一个名字。它找不到一个行为。你从廉价的语法检查迁移到更贵的语义检查,并接受一个事实 —— 那些廉价的检查一直在对你撒谎。

## 这个谎言在 smart agent 找到缝隙之前是隐形的

清单写"不能 lodash" —— agent 引入了 `underscore`。清单是名字。行为是"不要 utility library"。agent 满足了第一条,打破了第二条,测试套件通过了 —— 因为测试套件检查的也是名字。

如果你的不变量能通过机械验证,但系统仍在 drift,那是因为**名字不是行为**。停止写名字。
