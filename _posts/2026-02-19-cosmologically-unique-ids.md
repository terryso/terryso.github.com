## 引言

人类是一个探索性的物种，我们的探测器刚刚飞出太阳系，但也许有一天，我们会回头把我们所在的银河系称为"第一个"。在探索宇宙的道路上，有许多问题需要解决，今天我们要看的是其中一个很小的问题：**如何为设备（或任何对象）分配保证永远唯一的 ID？**

能够识别对象是构建其他协议的基础工具，它支撑着制造业、物流、通信和安全等领域。每艘飞船和卫星都需要 ID 来进行交通控制和维护历史记录。每个无线电、路由器和传感器都需要 ID，这样数据包才有源和目的地。每个制造组件都需要 ID 以实现可追溯性。在大规模场景下，数量会爆炸式增长：成群的机器人、数万亿个零部件，以及在一个文明供应链中流动的成千上万的货运集装箱。

ID 的一个关键功能是区分不同的对象，因此我们需要确保不会分配相同的 ID 两次。当我们试图在宇宙尺度上解决这个问题时，唯一 ID 分配就变成了一个更具挑战性的问题。

但我们可以尝试。

## 方案一：随机数（Random）

第一个也是最简单的解决方案是：每次设备需要 ID 时，选择一个随机数。

这非常简单，可能是最好的解决方案；你可以在任何时间、任何地点执行此操作，无需中央权威或任何形式的协调。

**主要问题是**：两个设备有可能碰巧选择相同的 ID。幸运的是，我们可以完全控制随机数的大小，从而控制碰撞的概率。这意味着我们可以将碰撞的可能性降至"实际上为零"。

你可能会说"实际上为零"还不够，虽然概率很小，但并不是"真正"为零，所以你仍然担心。但考虑这个例子：你现在被陨石击中的概率很小但非零，你甚至可能称之为"合理"（虽然有点偏执）的担忧。但是，你担心地球上每个人现在都被陨石击中吗？这个概率也是非零的，但它如此微乎其微，以至于我们将其视为不可能。这就是我们可以将 ID 碰撞概率降到多小的程度。

那么，这个概率需要多小我们才能放心呢？换个角度来看问题会有帮助：在预期发生碰撞之前，我们可以生成多少个 ID？

最新版本的通用唯一标识符（UUID）使用了 122 个随机位。使用生日悖论，我们可以计算出预期发生碰撞前的 ID 数量约为 2^61。

这算高还是低？这足够支撑人类在银河系扩张直到宇宙热寂吗？让我们通过观察宇宙的物理极限来计算一个有原则的数字。

### 宇宙极限

论文《Universal Limits on Computation》计算出，如果整个宇宙是一台最高效的计算机（被称为 computronium），在宇宙热寂之前它的操作上限是 10^120 次。如果我们假设每次操作生成一个新 ID，那么我们可以计算出需要多大的 ID 空间才能在宇宙耗尽时间之前避免碰撞。

使用生日悖论的近似，n 个随机数在 d 个值的集合中的碰撞概率为：

p(n, d) ≈ 1 - e^(-n(n-1)/2d)

我们希望 n = 10^120 个数字时碰撞概率为 p = 0.5（这是碰撞"预期"发生的近似值），所以我们可以解出 d：

d ≈ -n(n-1)/(2 × ln(1-p)) = -10^120(10^120-1)/(2 × ln(1-0.5)) ≈ 10^240

这就是如果我们想在宇宙热寂之前避免碰撞所需的 ID 空间大小。以位数计算，这需要 log₂(10^240) = 797.26，即至少 798 位。

这是最极端的上限，有点过度了。使用 798 位，我们可以为历史上的一切分配 ID，而且永远不会有预期碰撞。每个设备、每个芯片、每个芯片的每个组件、每次按键、每个时钟的每次跳动、每颗星星和每个原子，一切都可以使用这个协议被 ID 化，我们仍然不会预期碰撞。

### 合理的限制

一个更合理的上限可能是假设可观测宇宙中的每个原子获得一个 ID（我们假设原子不会被多次分配 ID，这是一个让步）。据估计，宇宙中有 10^80 个原子。使用相同的方程，我们发现需要 532 位才能在（概率上）避免到那一点的碰撞。

或者也许我们将宇宙的所有质量转换为 1 克的纳米机器人？我们将有 1.5 × 10^56 个机器人，这将需要 372 位的 ID。

现在我们有四种 ID 大小可以选择，取决于我们有多偏执：

- **798 位**：基于 computronium 极限
- **532 位**：基于原子数量
- **372 位**：基于 1 克纳米机器人
- **122 位**：来自 UUID 标准

> 注意，这假设在生成随机数时使用真正的随机性，但这有时是一个挑战。许多随机数生成器会使用带有非随机种子的伪随机数生成器。你需要确保你的硬件能够引入真正的随机性，例如来自量子源，或者使用加密安全的伪随机数生成器（CSPRNG）。如果那不可用，使用传感器数据、时间戳或其他非确定性来源可以帮助增加额外的随机性，但它不会是纯粹的随机性，因此会增加 ID 碰撞的概率。禁止任何"常见"的 ID 可能是个好主意，例如来自每个知名伪随机生成器的前 1,000 个 ID、全零 ID、全一 ID 等。

但是，如果我们异常偏执，**要求** ID 理论上保证唯一呢？不要这些概率上的胡说八道。这将带我们踏上一段旅程。

## 方案二：确定性方案（Deterministic）

### 中心化计数器

像往常一样，让我们从最简单的解决方案开始。

让我们创建一个使用计数器分配 ID 的单一中央计算机。当有人请求 ID 时，它分配其计数器的当前值，然后递增计数器，这样下一个 ID 将是唯一的。这个方案很好，因为它保证唯一性，而且 ID 的长度增长尽可能慢：对数增长。

如果所有 1 克纳米机器人从这个中央计算机获得 ID，最长的 ID 将是 log₂(1.5 × 10^56) = 187 位。实际上，由于编码可变长度值时的开销，它会稍微长一点。我们暂时忽略这一点。

好吧，这个解决方案有严重的问题。我看到的主要问题是访问。如果你在一个遥远的行星上，无法与中央计算机通信怎么办？或者你的行星离计算机太远，获取 ID 需要几天时间。无法接受。

### Dewey 方案

为了解决这个问题，我们可能会开始向各个方向发送可以分配唯一 ID 的卫星。想象一下，我们发送第一颗卫星 ID 为 0，然后下一颗为 1，继续递增。现在人们只需要从最近的卫星请求 ID，他们会得到类似 A.B 的 ID，其中 A 是卫星的 ID，B 是卫星上的计数器。例如，第四颗卫星分配其第十个 ID 将发出 3.9。这确保每个 ID 都是唯一的，而且获取 ID 更加容易。

但为什么停在卫星？为什么不让任何有 ID 的设备都能够分配新 ID？

例如，想象一艘殖民飞船被建造并从卫星 13 获得第六个 ID，所以它现在的 ID 是 13.5。殖民者将这艘船带到外缘，太远而无法与任何人通信。当他们到达他们的行星时，他们建造需要新 ID 的建筑机器人。他们无法从卫星请求 ID，因为他们太远了，但他们可以从他们的船请求 ID。建筑机器人获得 ID 13.5.3 和 13.5.4，因为船在此之前已经分配了 3 个 ID，其计数器在 3。现在这些机器人也可以分配 ID！

让我们称这个命名方案为 Dewey。

Dewey 与随机 ID 在所需位数方面相比如何？

如果 ID 的形式是 A.B....Z，我们可以使用 Elias omega 编码来编码它。目前我们忽略编码的小开销，假设每个数字使用其二进制值完美表示，但我们稍后会加回来。这意味着 ID 4.10.1 将有二进制表示 100.1010.1，有 8 位。我们可以看到 ID 中的每个值对数增长，因为计数器对数增长。

ID 随时间如何增长将取决于 ID 分配的顺序。

- 如果每个新设备都去原始设备，创建一个扩展的子树，那么 ID 将对数增长。这正是我们之前考虑的中央计算机模型。
- 如果我们采取另一个极端，每个新设备从最近的设备请求 ID，那么我们形成一个链。在这种情况下，ID 将线性增长。
- 或者，如果每个新设备选择一个随机设备来请求 ID，增长应该介于线性和对数之间。

**Dewey 方案的特点**：
- 最佳情况：每个节点使其子节点数量翻倍，然后重复。这导致它很快变宽。
- 最坏情况：链式结构导致线性增长。

### Binary 方案

这是 ID 分配方案的另一次尝试，让我们看看它是否会增长得更慢。

将整个 ID 空间可视化为二叉树。每个设备将在此树上的某处有一个 ID。为了分配新 ID，设备将取其下方的列（每台设备的列从左或右交替）并分配该列中的 ID。

**Binary 方案的特点**：
- 子树和链情况都线性增长
- 最佳情况树在所有方向上大致相等地增长
- 比较适合新节点同样可能从旧节点和新节点请求的增长模型

### Token 方案

让我们尝试逆向设计一个可以为链树对数增长的方案。

我们知道计数器对数增长，所以理想情况下，ID 在添加新节点时只会递增计数器。

一个想法是有一个带有跳数计数的令牌传递给子节点。但是，当设备收到新的 ID 请求并且它没有令牌可以传递时会发生什么？我们将有一个令牌索引，每次父节点必须创建新令牌时递增。新令牌将被追加到父 ID。

**Token 方案的特点**：
- 链式结构对数增长
- 但只要再增加一个子节点，方案就开始线性增长
- 如果我们的图在深度和宽度上一起增长，我们就回到了线性状态

### 数学证明：最坏情况必然是线性增长

不幸的是，我们无法设计出一个在最坏情况下总是对数增长的方案。以下是证明。

为了证明任何方案必须增长多快，我们将查看随着节点添加，可能的 ID 数量增长多快。这将需要遍历每个可能的分配历史，然后计算所有可能分配历史空间中有多少个唯一的可能 ID。

重要的是要注意，每条路径必须产生不同的 ID。如果任何两条路径产生相同的 ID，那意味着可能生成两个具有相同 ID 的节点。

一般来说，注意每次我们添加新 ID 时，我们向所有可能路径的树中的每个节点添加一个新叶。这意味着我们需要考虑的 ID 数量增长为 2^(n-1)，对于 n 个节点。

这是一个问题。即使在理想情况下，我们使用计数器标记所有可能历史空间中的每个可能节点（这实际上是一个有效的 ID 分配方案，并生成我们已经看到的 2-Adic Valuation 方案），计数器的内存对数增长。无论我们使用什么方案，内存必须至少按 log₂(2^(n-1)) = n-1 的顺序增长，即线性。

## 空间定居模型

让我们考虑不同的模型来近似人类如何扩展到宇宙中。

### 小规模

第一个也是最简单的模型是随机父节点选择。每次添加设备时，它将随机从所有以前的设备中选择以请求 ID。这将产生所谓的随机递归树。

在 2,048 个节点的小规模下，使用 Elias omega 编码：
- **最佳方案**：Binary，其次是 Dewey，Token 最差
- 随机树将以大致相等的速率在深度和宽度上增长，这对 Binary 是最佳情况

对于优先连接随机图（节点更有可能连接到有更多连接的节点）：
- **最佳方案**：Dewey，其次是 Token，Binary 差距很大
- 树的宽度将主导深度

对于适应度模型（每个节点初始化时有一个决定其受欢迎程度的适应度分数）：
- **结果**：Dewey 和 Binary 表现相当，Token 产生最差的 ID

### 中规模

我们为每个增长模型运行 1,000 次模拟，构建多达约一百万（2^20）个节点的图。我们绘制图中最大 ID 随时间的变化。

**结果**：大多数图表上看到大致直线（除了 Preferential 增长模型的 Binary 和 Fitness 增长模型，它们弯曲少量）。直线是 ID 增长实际上是对数的强烈指示。

对数增长的原因：
- 在随机增长模型中，每个节点在统计上与其他节点无法区分，所以我们应该期望每个节点随时间看到相同的平均子树。
- 在分布上，根下的子树应该看起来类似于第一百万个节点下的子树，只是在较小的规模上。
- 这表明我们可以使用这些子树之间的递归关系来推断整体缩放定律。

### 大规模：宇宙级扩展

为了确定这些方案对于宇宙范围的人类可能需要多少位，我们需要评估 ID 在世界之间如何增长的模型。

我们将使用 Fitness 增长模型的百万节点模拟来模拟行星表面头几年的 ID 分配。

**关键假设和计算**：
- 银河系中可能有约 400 亿颗宜居行星
- 可观测宇宙中大约有 2 万亿个星系
- 如果假设行星在银河系中接近均匀分布，银河系半径（以行星跳数为单位）约为 2121
- 如果每个行星在定居下一个最近的行星之前产生约 10 亿个 ID

**计算结果**（使用 Dewey 方案）：
- 达到银河系边缘时的 ID 长度：6.5534 × ln(10^9) × 2121 ≈ 288,048 位
- 星系之间的跳数：约 7816
- 最终 ID 长度：288,048 × 7816 = 2,251,383,168 位
- **仅存储 ID 就需要约 281.4 MB 内存**

**与随机方案比较**：
- 这个确定性方案与随机方案相比太糟糕了
- 随机方案即使在最偏执的情况下也只使用 798 位

## 结论

到目前为止，对于通用唯一 ID，最安全的选择似乎是大范围随机数，使碰撞概率实际上为零。但考虑如何将这个概率降到真正为零是有趣的：设计不同的 ID 分配方案、运行模拟以及建模人类在宇宙中的扩张。

**关键要点**：

1. **随机方案是最佳选择**：使用 798 位的随机 ID 可以覆盖到宇宙热寂为止的所有可能操作，碰撞概率实际上是零。

2. **确定性方案的理论局限**：无论设计什么样的确定性 ID 分配方案，在最坏情况下必然是线性增长的。

3. **不同方案适用于不同场景**：
   - Dewey：适合宽度主导的增长模型（如优先连接网络）
   - Binary：适合深度和宽度均衡增长的情况（如随机网络）
   - Token：理论上对链式结构好，但实际应用受限

4. **实践建议**：除非你有极端的确定性要求，否则使用大范围的随机 ID（如 UUID）是最好的选择。

5. **安全考虑**：可以使用公钥作为 ID（随机方案）或签名链（确定性方案）来防止 ID 欺骗。
