## 突然的寂静

2026年1月14日，大约21:00 UTC，互联网的底层管道发生了某种变化。GreyNoise 全球观测网格记录到全球 telnet 流量的突然、持续崩溃——这不是渐进式下降，不是扫描器流失，也不是数据管道问题，而是一个**阶跃函数（step function）**。

前一个小时：约74,000个会话
下一个小时：约22,000个会话
再下一个小时：降至约11,000个会话，并稳定在这个水平

六天后，1月20日，CVE-2026-24061 的安全公告发布到 oss-security 邮件列表。1月26日，CISA 将其加入 KEV（已知已利用漏洞）目录。

我们在1月22日写过关于漏洞公开后前18小时的利用活动。但本文要讨论的是另一件事：在 CVE 公开**之前**发生的全球 telnet 流量结构性变化，以及为什么我们认为这两件事可能并非独立事件。

---

## 断崖式下跌

从2025年12月1日到2026年1月14日，GreyNoise 观察到平均每天约 **914,000** 个不可伪造的 telnet 会话（总计5,120万个会话）——我们称之为"基线"。

1月14日21:00 UTC，小时流量在一个时间刻度内**暴跌65%**。两小时内，已低于基线83%。新的平均值稳定在每天约373,000个会话——**持续减少59%**，这种状态一直持续到撰写本文时（2月10日）。

这不是逐渐减少。拐点附近的小时数据说明了一切：

| 时间 (UTC) | 每小时会话数 | 备注 |
| --- | --- | --- |
| 1月14日 19:00 | 73,900 | 正常基线 |
| 1月14日 20:00 | 64,722 | 正常基线 |
| **1月14日 21:00** | **22,460** | **一小时内暴跌65%** |
| 1月14日 22:00 | 11,325 | 低于基线83% |
| 1月14日 23:00 | 11,147 | 新底线确立 |
| 1月15日 00:00 | 12,089 | 维持在降低水平 |

这种阶跃函数——在单小时内传播——读起来更像是路由基础设施的配置变更，而非扫描群体的行为漂移。

---

## 谁沉默了

18个 ASN（在下跌前有显著 telnet 流量，每个超过5万会话）在1月15日后归零。一些引人注目的名字：

- **Vultr** (AS20473) — 下跌前38.2万会话，然后消失
- **Cox Communications** (AS22773) — 15万会话，消失
- **Charter/Spectrum** (AS20115) — 14.1万会话，消失
- **BT/British Telecom** (AS2856) — 12.7万会话，消失

**五个国家**完全从 GreyNoise 的 telnet 数据中消失：津巴布韦、乌克兰、加拿大、波兰和埃及。不是减少——是**归零**。

与此同时，主要云服务商基本未受影响，甚至有所增长：
- AWS **增长78%**
- Contabo **增长90%**
- DigitalOcean 基本持平，+3%

云服务商在主要 IXP 拥有广泛的私有对等互联（private peering），可以绕过传统的传输骨干网路径。住宅和企业 ISP 通常没有。

## 过滤器在哪里？

这种模式指向一个或多个北美一级传输供应商实施了端口23过滤：

时间点——21:00 UTC，即16:00 EST——与美国维护窗口一致。美国住宅 ISP（Cox、Charter、Comcast 下降74%）遭受重创，而同一大陆上的云服务商通过绕过变更的对等互联保持正常。Verizon/UUNET (AS701) 下降79%，作为主要一级骨干网，这符合它是过滤实体或直接位于过滤实体上游的情况。

AS701 上剩余的21%流量代表不经过过滤链路的路径。

依赖跨大西洋或跨太平洋骨干网路由来访问美国托管基础设施的国家受打击最重。拥有强大欧洲直接对等互联的国家（法国+18%，德国-1%）基本未受影响。

中国骨干网提供商（中国电信和中国联通）都下降了约59%，非常均匀。这种均匀性表明过滤器位于跨太平洋链路的美国一侧，而非中国境内。如果这是中国防火墙的行动，我们预期中国运营商之间的影响不对称，并且会有更硬性的切断。

## 然后CVE来了

CVE-2026-24061 是 GNU Inetutils telnetd 中的一个**严重（CVSS 9.8）身份验证绕过漏洞**。该缺陷是 telnetd 在 telnet 选项协商期间处理 `USER` 环境变量时的参数注入。攻击者发送 `-f root` 作为用户名值，`login(1)` 乖乖跳过身份验证，交出 root shell。

无需凭据。无需用户交互。该漏洞代码在2015年的一次提交中引入，潜伏近11年未被发现。

时间线：

| 日期 | 事件 |
| --- | --- |
| 1月14日 21:00 UTC | Telnet 骨干网下跌开始 |
| 1月20日 | CVE-2026-24061 公告发布到 oss-security |
| 1月21日 | NVD 条目发布；GreyNoise 标签部署；首次利用被观测到 |
| 1月22日 | GreyNoise 发布关于初始18小时利用活动的文章 |
| 1月26日 | CISA 将 CVE-2026-24061 加入 KEV 目录 |

telnet 下跌与公开 CVE 披露之间的**六天间隔**是最有趣的部分。表面上，下跌不可能是由 CVE 披露"引起"的，因为下跌先发生。但"引起"并不是唯一值得考虑的关系。

## 推测

负责任的披露时间线不是从发布开始的。根据公开来源，发现此漏洞的研究员（署名为 Kyu Neushwaistein / Carlos Cortes Alvarez）在1月19日报告了该缺陷。但是，导致补丁准备就绪、公告起草、CISA 准备在发布后六天内将其加入 KEV 的协调工作通常早于披露前一天就开始。

以下是我们认为可能发生的情况：关于一个**极易利用、未经身份验证的 root 访问漏洞**影响 telnet 守护进程的**提前通知**，传达给了能够在基础设施级别采取行动的各方。某骨干网或传输供应商——可能响应协调请求，可能根据自己的评估采取行动——在传输链路上实施了端口23过滤。

过滤于1月14日生效。公开披露于1月20日跟进。

这可以解释：

- 时间间隔（提前通知 → 基础设施响应 → 公开披露）
- 过滤的特异性（端口23/TCP，而非一般路由变更）
- 影响拓扑（依赖传输的路径受影响，直接对等互联的路径不受影响）
- 持续性质（数周后过滤器仍在实施）

我们无法证明这一点。骨干网下跌可能完全是巧合——ISP 多年来一直在慢慢迈向过滤遗留的不安全协议（参考：Wannacry），而1月14日可能只是某人的变更控制单终于被执行的时间。相关性、时间接近性和合理机制**绝不等于**因果关系。

但是，一级骨干网实施似乎是端口23过滤的措施，六天后是易于利用的 root 访问 telnet 漏洞的披露，四天后是 CISA KEV 列表，这一组合值得记录和考虑。

## 下跌后的世界

1月14日之后的 telnet 格局显示出周期性的锯齿模式——周期性峰值后跟低谷（例如，1月28日80.6万会话，然后1月30日19.1万会话）。这可能表明间歇性过滤器应用、过滤基础设施周围的路由抖动，或恰好使用不受过滤器影响的路径的扫描器活动。

周平均值讲述了持续的故事：

| 起始周 | 平均每日会话 | 占基线百分比 |
| --- | --- | --- |
| 12月1日 | 1,086,744 | 119% |
| 1月5日 | 985,699 | 108% |
| 1月19日 | 363,184 | 40% |
| 1月26日 | 407,182 | 45% |
| 2月2日 | 322,606 | 35% |

我们现在运行的流量大约是下跌前基线的三分之一，趋势仍然略有下降。

## 实际影响

如果你在任何地方运行 GNU Inetutils telnetd——考虑到11年的时间窗口，仍有大量嵌入式系统、网络设备和遗留 Linux 安装可能存在它——请打补丁到版本2.7-2或更高版本，或完全禁用该服务。

联邦机构的 CISA KEV 修复截止日期是2026年2月16日。如前所述，GreyNoise 观察到披露后数小时内就出现利用尝试，活动在2月初达到每天约2,600个会话的峰值，然后逐渐减弱。

如果你是网络运营商，尚未在边界过滤端口23，我们在此记录的骨干网级别过滤表明，无论如何，行业都在向那个方向发展。互联网传输基础设施的重要部分的上游似乎有人决定了 telnet 流量不再值得承载。

这可能是正确的决定。

## 总结

这篇文章揭示了一个有趣的现象：在全球范围内，telnet 流量在短时间内暴跌了近60%，而六天后才公开披露了一个严重的 telnetd 漏洞。

**核心要点：**

1. **基础设施级别的防护**：大型网络运营商可能在漏洞公开前就主动实施了端口过滤，这是一种"先发制人"的防护措施

2. **云服务商的优势**：由于拥有广泛的对等互联网络，云服务商受到的影响远小于传统 ISP

3. **遗留协议的风险**：Telnet 这种明文、无身份验证的协议早已应该被淘汰，这次事件可能加速了它的消亡

4. **负责任披露的力量**：如果这确实是协调行动的结果，它展示了安全研究界、政府机构和基础设施运营商之间协调的重要性

5. **防御深度的必要性**：对于运行关键服务的组织来说，依赖单一防护层是不够的——即使有漏洞，网络层面的过滤也能提供额外保护

作为开发者，我们应该从这个案例学到：
- 尽可能使用加密、强身份验证的协议（如 SSH）
- 及时更新和修补遗留系统
- 在网络边界实施最小权限原则
- 关注安全公告，但也要提前做好基础防护
